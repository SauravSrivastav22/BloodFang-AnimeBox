 #1  [OURS] HIGH — Hosted build hits AniList rate limit (429); no cache
     >>> STATUS: FIXED & DEPLOYED 2026-07-15 <<<
     Added client-side cache (memory + sessionStorage, survives reloads),
     in-flight de-dupe, concurrency cap (4), and 429 retry honoring
     Retry-After — in src/anilist-direct.js gql(). Verified in the live
     bundle. Reloads now refetch nothing; bursts throttle + auto-retry.
--------------------------------------------------------
WHAT: AniList's rate limit is currently 30 requests/minute (confirmed via
      response header `X-RateLimit-Limit: 30`). In a burst test we got HTTP 429
      after ~10 requests (Retry-After ≈ 31s).
WHY IT BITES US: the home page fires ~10 AniList requests on FIRST load
      (trending grid + "Trending this Season" row + 8 genre rows). A browser
      reload re-fires all of them; add a couple of searches / detail opens and
      the user blows past 30/min and starts getting 429s.
ROOT CAUSE: the LOCAL build has a server-side cache (server/index.js `cached()`)
      that collapses repeat AniList calls. The STATIC/hosted build talks to
      AniList directly (src/anilist-direct.js) with **no cache and no throttling**,
      so every render is a fresh hit.
SYMPTOMS ON THE LIVE SITE:
  - Home rows randomly disappear (see #2).
  - Grid shows the "Couldn't load anime" error screen intermittently.
  - Opening a detail page sometimes shows "Couldn't load this title".
REPRO: hard-refresh the live site 2–3 times quickly, or open several titles in a
      row → some rows vanish / error screens appear.
SUGGESTED FIX:
  - Add a client-side cache in anilist-direct.js (in-memory + sessionStorage,
    short TTL) mirroring the server cache — biggest win.
  - De-dupe in-flight identical requests.
  - Throttle concurrency (e.g. max 3–4 in flight) and/or stagger the home rows.
  - On 429, respect `Retry-After` and retry instead of failing.

 #2  [OURS] MED — Genre/season rows vanish silently on any fetch error
     >>> LARGELY MITIGATED by #1's 429 retry (rows no longer drop on rate
         limits). Still worth a retry/placeholder for genuine failures. <<<
--------------------------------------------------------
WHAT: In App.jsx `Row`, any fetch failure (incl. a 429 from #1) sets
      `failed=true` and the component returns null — the whole row just
      disappears with no message.
IMPACT: on the live site the home page can show a different set of rows each
      load; looks broken/random. Compounds #1.
REPRO: trigger a 429 (see #1) → one or more home rows are simply gone.
SUGGESTED FIX: show a small "couldn't load — retry" placeholder instead of
      hiding, and/or retry on 429. (Fixing #1 makes this rare.)

 #4  [OURS] MED — Hosted build always offers/defaults to Dub, even for sub-only
     >>> STATUS: FIXED & DEPLOYED 2026-07-15 <<<
     Hosted build now starts on SUB by default (dub unverified), so sub-only
     titles play immediately instead of a blank dub player. The Dub button is
     still shown for titles that have one. (EmbedPlayer `dubUnverified` prop,
     driven by the DIRECT build flag from DetailPage.)
--------------------------------------------------------
WHAT: The dub check needs the HiAnime scraper (server-only). The static build's
      getDub() returns `{dubAvailable:null}`, so the player treats dub as
      available and (per Settings default) starts on Dub. For a sub-only title
      the Dub embed can load blank.
IMPACT: on the live site, sub-only shows may open to a blank/again player until
      the user switches to Sub.
REPRO: [NEEDS BROWSER] open a sub-only title on the live site, press play.
SUGGESTED FIX: in direct mode, default audio to Sub (safer), OR detect via one
      of the embed providers, OR just surface the Sub/Dub toggle more prominently.

 #9  [OURS] LOW — Movies/specials can show a bogus extra "episode"  (QA round 2)
     >>> STATUS: FIXED & DEPLOYED 2026-07-15 <<<
     For MOVIE/MUSIC/SPECIAL with a known `episodes`, we now trust that count
     (no max() with the padded streamingEpisodes) and cap the rich-tile render
     at totalEps. Long ongoing TV still uses max() (see #6).
--------------------------------------------------------
WHAT: For a few single-unit titles AniList's `streamingEpisodes` has MORE
      entries than the real episode count (e.g. Your Name / Kimi no Na wa:
      episodes=1 but streamingEpisodes=2 — the 2nd entry is a PV/teaser). Our
      count = max(episodes, richLen), so the detail page shows 2 episodes for a
      1-episode movie; clicking "Episode 2" opens a player that has no such
      episode → blank.
IMPACT: minor/cosmetic; only affects some movies/specials.
REPRO: open a movie whose AniList streamingEpisodes list is padded → an extra
      episode tile appears.
SUGGESTED FIX: when format is MOVIE/MUSIC/SPECIAL and `episodes` is known, cap
      totalEps at `episodes` (don't max with richLen). Keep max() only for
      ongoing TV where episodes is null.
